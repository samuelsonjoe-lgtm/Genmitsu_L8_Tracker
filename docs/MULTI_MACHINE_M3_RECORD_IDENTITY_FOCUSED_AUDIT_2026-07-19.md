## 4. Optional-ID normalization

`normalizeOptionalMachineId(value)` (`index.html:547-551`):
```js
function normalizeOptionalMachineId(value) {
  if (typeof value !== 'string') return '';
  const id = value.trim();
  return id || '';
}
```
Verified: only trimmed non-blank strings survive (non-string input, including numbers/objects/arrays/`null`/`undefined`, → `''`); **no ID is ever generated** by this function (it is pure normalization, never calls `uid()`); every call site (`normalizeMaterialTests`, `normalizeProductionEvidence`, `normalizeProductionSetting`, `promotionMachineSnapshot`, `gridMachineSnapshot`, `activeOwnedMachineSnapshot`, `promotionEvidenceSnapshot`, `buildGridCellMaterialTestRecord`, `buildWizardMaterialTestRecord`, `normalizeProject`) only ever *reads* an already-present `machineId`-shaped value from its input — none infers an ID from `label`/`key`/`manufacturer`/`model`/the active machine. Existing (pre-M3) records remain ID-free after normalization because a missing source value normalizes to `''`, and every writer uses `...(id ? {machineId:id} : {})` to omit the key entirely rather than writing an empty string (confirmed by the M3 fixture: "Historical label/key records receive no backfilled machineId" — live-equivalent source check: `!legacy.machineId`). Normalization does not mutate its input (all normalizers build new object literals via spread, never assign back onto the parameter). Opening/rendering a record does not call any of these normalizers with a *different* value than what was stored — no backfill path exists. Editing unrelated fields preserves the `machineId` because every editor round-trips the existing `machine`/`machineId` sub-structure unchanged unless the user explicitly changes the machine field (confirmed live for Projects, §13; confirmed live for Grids, §6).

## 5. Active-machine snapshot

`activeOwnedMachineSnapshot()` (`index.html:552-556`):
```js
function activeOwnedMachineSnapshot() {
  const id = normalizeOptionalMachineId(state.activeMachineId);
  const machine = id ? machineById(id, state.machines) : null;
  return machine ? { machineId:id, key:'custom', label:ownedMachineDisplayName(machine) } : null;
}
```
Verified: a valid active owned machine returns a stable `machineId` (the machine's own ID, never regenerated); `key:'custom'` is always used, which is a member of the pre-existing `productionMachineKeys` allowlist (`['genmitsu-l8-20w','genmitsu-l8-40w','custom']`), so this remains compatible with every existing allowlist-gated code path (`gridMachineSnapshot`'s `!productionMachineKeys.includes(key)` guard, `normalizeProductionSetting`'s `productionKnownValue` allowlist check); `label` is frozen from `ownedMachineDisplayName(machine)` at snapshot-construction time (a plain string copy, not a live reference — confirmed no getter/live-binding exists); a blank owned-machine name correctly produces "Unnamed workshop machine" (via `ownedMachineDisplayName`'s own fallback, live-confirmed in the M3 fixture: "Blank owned name uses the unnamed workshop fallback"); the Genmitsu Reference fallback is never used here (this function returns `null`, not a Genmitsu-labeled object, when there is no valid active machine — the *caller*, `activeMachineSnapshot()` with zero arguments, is what falls back to Genmitsu wording, and only when `activeOwnedMachineSnapshot()` itself returned `null`); the raw ID is never shown as a label (label is always `ownedMachineDisplayName`'s human-readable output). Invalid `activeMachineId` (missing machine) and an all-archived-machines state are both safe: `machineById(id, state.machines)` returns `null` for a non-existent ID, and `machineById` performs no archived-filtering itself, so a snapshot request against an *archived* active machine would still theoretically resolve — but this can't arise in practice because M2's `resolveActiveMachineId`/`synchronizeMachineIdentity` (unchanged, called on every mutation) never leaves `activeMachineId` pointed at an archived machine. No owned machine is ever fabricated by this function (it only reads `machineById`, never constructs a `machines` entry).

`activeMachineSnapshot(active)` (`index.html:597-598`):
```js
function activeMachineSnapshot(active) {
  if (arguments.length === 0) return activeOwnedMachineSnapshot() || activeMachineSnapshot(state.machineProfile);
  ...
}
```
This preserves **every existing explicit-argument call site exactly as before** (`activeMachineSnapshot('20W')`, `activeMachineSnapshot('40W')`, `activeMachineSnapshot(machineObject)` are all untouched, since the `arguments.length === 0` branch only fires for zero-argument calls). Only the **zero-argument** call semantics changed: previously defaulted silently to `state.machineProfile` (the Reference profile); now prefers the real owned active machine when one exists, falling back to the Reference-profile-derived snapshot only when it doesn't. This is a deliberate, minimal, backward-compatible upgrade — confirmed by inspecting all remaining call sites (§6 for the Grid form; `openProductionSetting`'s new-setting branch, §8; `openMaterialTestWizard`, §7).

## 11. Promotion matching findings — the highest-risk boundary, verified exhaustively

`productionMachineIdentityMatches(candidateMachine, targetMachine)` (`index.html:7512-7517`):
```js
function productionMachineIdentityMatches(candidateMachine, targetMachine) {
  const candidate = candidateMachine || {}, target = targetMachine || {};
  const candidateId = normalizeOptionalMachineId(candidate.machineId), targetId = normalizeOptionalMachineId(target.machineId);
  if (candidateId || targetId) return !!candidateId && !!targetId && candidateId === targetId;
  return !!candidate.key && !!target.key && candidate.key === target.key;
}
```

**Complete truth table independently derived and verified** (both by hand-tracing this exact function and by the passing M3 fixture, which directly exercises four of these branches):

| Scenario | Candidate | Target | Result | Correct? |
|---|---|---|---|---|
| Both IDs, same ID | `machineId:'A'` | `machineId:'A'` | **true** | ✓ exact-ID match required, satisfied |
| Both IDs, different ID (Machine A evidence vs Machine B candidate) | `machineId:'A'` | `machineId:'B'` | **false** | ✓ cross-machine leakage correctly blocked |
| Same label, different ID | `machineId:'A', label:'X'` | `machineId:'B', label:'X'` | **false** (label is not even inspected by this function) | ✓ same-name/different-ID does not falsely match |
| Same ID, renamed label | `machineId:'A', label:'Old'` | `machineId:'A', label:'New'` | **true** (label irrelevant to this function) | ✓ same-ID-with-renamed-label still matches |
| Exactly one side has an ID | `machineId:'A'` | *(no `machineId`)* | **false** | ✓ one-sided ID never counts as an exact match — no automatic cross-boundary proof is granted |
| Neither side has an ID, same legacy key | `key:'genmitsu-l8-20w'` | `key:'genmitsu-l8-20w'` | **true** (falls through to legacy key comparison) | ✓ existing legacy 20W-vs-20W promotion behavior fully preserved |
| Neither side has an ID, different legacy key | `key:'genmitsu-l8-20w'` | `key:'genmitsu-l8-40w'` | **false** | ✓ preserves the pre-existing legacy exact-key requirement (unchanged from before M3) |

Live-verified directly via the M3 fixture's own assertion (independently re-derivable from source, not merely trusted): `productionMachineIdentityMatches({key:'custom',machineId:'m3-a'},{key:'custom',machineId:'m3-a'})` → true; `...{machineId:'m3-b'})` → false; `...{key:'custom'})` (no ID) → false; `productionMachineIdentityMatches({key:'custom'},{key:'custom'})` (neither has an ID) → true.

**Every call site correctly upgraded to this single function, confirmed by direct source read (no split-brain between soft-validation and hard-save gating):**
- `productionValidSelectedFields`'s machine-dependent-field gate: `if (fieldData.machineDependent && target?.machine && !productionMachineIdentityMatches(candidate?.machine, target.machine)) { blocked.push(key); return; }`
- `promotionValidateAction`'s mismatch warning: `if (targetSetting && !productionMachineIdentityMatches(machine, targetSetting.machine)) warnings.push('Machine identity mismatch: machine-dependent values are blocked.')`
- `savePromotionTransaction`'s hard save-time block: `if (targetSetting && !productionMachineIdentityMatches(candidate.machine, targetSetting.machine) && ...) return { ok:false, errors:['Machine-dependent fields cannot be promoted across machines.'] };`
- `productionSettingsCanSupersede`'s supersede-eligibility check: `productionMachineIdentityMatches(oldSetting.machine, replacement.machine)` (replacing the old raw `.key ===` comparison).

**No cross-machine proof leakage and no unintended loss of valid legacy promotion was found.** Archived-machine evidence and unknown-`machineId` evidence are handled identically to any other evidence by this matching function (it only ever inspects the `machineId`/`key` strings already present on the snapshot — archival status of the *current* `machines` collection entry is irrelevant to matching a frozen historical snapshot). Legacy custom-alias matching (the *separate*, pre-existing `projectWizardMachineCompatibility` label-alias mechanism used only by the Project Wizard's own evidence-ranking display, not by Library promotion) is untouched by M3 — see §7 of the architecture review and the scope note below.
