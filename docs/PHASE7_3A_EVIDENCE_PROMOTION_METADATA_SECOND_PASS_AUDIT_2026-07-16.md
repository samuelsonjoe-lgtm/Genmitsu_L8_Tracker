## 2. Verified-date-only Update — independently confirmed

Traced the full path by hand: `capturePromotionForm()` computes `metadataSelections.verifiedDate = action==='create' || (action==='update' && !!targetSetting && promotionSession.verifiedDate !== String(targetSetting.verifiedDate||''))` (`index.html:5469`) — so changing only the date field (while the rendered form's other fields already match the target, per `syncPromotionMetadataFromTarget`'s fresh load) sets `verifiedDate:true` and leaves `status:false`. In `savePromotionTransaction()` (`index.html:5395`), `statusNeedsValidation = action==='create' || !targetSetting || metadataSelections.status===true` — since `status` is `false`, the full `promotionStatusRules()` (which demands `physicalChecks`/notes for `verified`) is **skipped entirely**; instead `dateErrors = action==='update' && metadataSelections.verifiedDate ? promotionVerifiedDateRules(requestedStatus, verifiedDate) : []` runs the much lighter check. This is exactly "no new physical verification check is required solely because the date changed on an already verified target." `mergePromotionIntoSetting()` (`index.html:5385`) writes `next.verifiedDate` when `action==='create' || metadata.status || (action==='update' && metadata.verifiedDate)` — the third disjunct fires here, correctly applying the new date while `metadata.status` staying `false` means nothing else about verification is touched. Preferred (`index.html:5384`, requires its own `metadata.preferred` flag), scope/batch/label (each gated on their own independent flag), unknown top-level/nested fields (never touched by any of these conditionals — they survive via the initial `promotionClone(target)`), and existing evidence (untouched, a new entry is only *appended*) all remain byte-identical. `promotionVerifiedDateRules(status, verifiedDate)` (`index.html:5358-5363`) — hand-verified all four requested invalid combinations: `('verified','')` → error (clearing date while verified); `('tested','2026-07-15')` → error (adding a date while non-verified); a malformed/whitespace-only date string fails the `.trim()` truthiness check identically to an empty one, so it's treated the same as "no date" and correctly blocked when status is `verified`. All of these run and block **before** any mutation (both `promotionEligibility`/`statusErrors`/`dateErrors` are checked and short-circuit-returned at `index.html:5396`, prior to any `promotionClone`-based transaction work).

## 3–5. Preferred true→false, false→true, and unchanged-across-switches — independently confirmed

`capturePromotionForm()`'s preferred dirty check (`index.html:5470`): `preferred = action==='create' || (action==='update' && !!targetSetting && promotionSession.preferred !== !!targetSetting.preferred)`. Unchecking an already-`true` target's Preferred box sets `preferred:true` (dirty) with `session.preferred===false` (the new intended value). In `savePromotionTransaction()`, `preferredRequested = action==='create' || (action==='update' && metadataSelections.preferred && session.preferred)` (`index.html:5420`) — note the **second** conjunct requires `session.preferred` itself to be `true`, so a true→false edit (`session.preferred===false`) makes `preferredRequested` false regardless of the dirty flag — **no conflict scan, no `confirmDemotion` call, ever, for a clearing edit.** `mergePromotionIntoSetting()`'s preferred-write (`index.html:5384`) still fires because it only checks `metadata.preferred` (not its direction), correctly writing `next.preferred = false`. For false→true, `preferredRequested` is true, `conflicts` is computed from the same `operation`+`machine.key` filter used throughout this feature (`index.html:5420`), and the accept/decline branches (`index.html:5421-5422`) are unchanged from what I already verified in the prior audit in this chain — decline sets only the target's own `preferred` to `false` and touches no conflicting record; accept demotes only the matched conflicts. Switching targets without touching the checkbox: `syncPromotionMetadataFromTarget()` reloads `next.preferred = !!targetSetting.preferred` and resets the flag to `false` on every switch (confirmed at `index.html:5177`, `5497-5499`), so returning to a previously-viewed target reloads its actual current state rather than a stale in-session value — this is precisely the mechanism that closes the target-switch defect from my prior audit, independently re-confirmed here by tracing all three change handlers (`targetProfileId`, `promotionAction`, `targetSettingId`) and `startPromotion()`'s evidence-only-startup sync path, all of which route through `syncPromotionMetadataFromTarget()`.

## 6. Evidence-only preservation — independently confirmed, with the same nuance Grok's audit also flagged

Traced `capturePromotionForm()` + `savePromotionTransaction()`'s evidence branch together: even if a tampered/scripted form submission set `metadataSelections.label`/`.measurementScope`/`.batchLabel`/`.status` to `true` (these four are **not** unconditionally zeroed for the evidence action the way `verifiedDate`/`preferred` explicitly are, at `index.html:5472-5475`), it cannot matter, because `savePromotionTransaction()`'s action-branch for `evidence` never calls `mergePromotionIntoSetting()` at all — `nextSetting = targetSetting` directly (`index.html:5414`), and only an evidence record is appended afterward. I independently reached the same observation Grok's audit also makes (its M1): today this is architecturally safe (the merge function that would read those four flags is simply never invoked on this path), but it is a latent inconsistency — four of six metadata flags rely on "the merge call never happens" for their safety, while two rely on an explicit zero-out, which is an easy asymmetry to accidentally break if Evidence-only's transaction branch is ever refactored to route through `mergePromotionIntoSetting()` for some other reason. I agree with Grok's characterization of this as Minor and hardening-only, not a live defect.

## 8. Save-time adversarial validation — independently confirmed

`savePromotionTransaction()` never trusts session state without re-deriving it: `promotionValidSelectedFields()` is recomputed from `candidate.fields` (not from whatever the form last rendered), `statusNeedsValidation`/`dateErrors` are recomputed from `metadataSelections` and `promotionStatusRules()`/`promotionVerifiedDateRules()` freshly, and all of `eligibility.errors`/`statusErrors`/`dateErrors` are checked and short-circuited (`index.html:5396`) before any `promotionClone`/mutation work begins. Hand-traced: `metadataSelections.verifiedDate=true` with an invalid/blank date and `requestedStatus:'verified'` → `dateErrors` non-empty → blocked, `state.profiles` untouched (the clone/mutation code never executes). A verified-date edit while `requestedStatus` isn't `'verified'` → also blocked by the same rule's second branch. `metadataSelections.preferred=true` during an `evidence` action → irrelevant, since `preferredRequested`'s definition (`index.html:5420`) requires `action==='update'` specifically — an `evidence` action can never set `preferredRequested` true regardless of what `metadataSelections.preferred` claims, so this crafted input has no effect, confirmed by the condition's structure alone. Preferred false→true with an unresolved conflict (`confirmDemotion` returns falsy) → target's own `preferred` is forced `false` in-clone (`index.html:5421`), no other record touched, and the *rest* of the Update still commits (this is intentional non-blocking behavior per the architecture spec, verified in the prior audit). Preferred true→false against a "stale target identity" (e.g. `targetSettingId` pointing at a setting that no longer exists) → `targetSetting` resolves to `undefined`, `promotionEligibility()` rejects with "Choose a target Production Setting" before any of the preferred logic is reached. Unknown `metadataSelections` keys (e.g. a crafted extra key not in the six recognized ones) are simply never read by any of the six explicit `metadata.X` checks in `mergePromotionIntoSetting()` — inert by construction, not explicitly rejected, but also not exploitable since nothing reads them.
