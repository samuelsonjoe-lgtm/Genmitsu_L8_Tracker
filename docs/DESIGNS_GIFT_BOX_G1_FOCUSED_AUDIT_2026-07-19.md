## 6. Lift-off-lid findings

**Structural (production) lift-off lid geometry is correct**: the lid is a plain rectangular panel (`giftBoxRectPanel('lid','LID',lidWidth,lidDepth)`) sized to the shell footprint (flush) or footprint+overhang (overhanging), matching the result-summary math (§3/§4). **However, the Finished View representation of the lift-off lid is critically broken — see §11.** A lift-off lid correctly has no hinge axis in the underlying data model (no hinge fields are computed for `lidType==='lift-off'`), and the Finished View's `state` parameter does distinguish "closed" vs "open" via different `lidX`/`lidY`/`lidScale` values intended to represent "lid to the side" (a non-hinged repositioning) rather than a rotation — this is conceptually the *correct* choice per the audit's own guidance ("do not show a lift-off lid rotating around the rear edge as if hinged"), but the implementation fails to actually render the lid shape at all (§11), so this correct conceptual choice is currently invisible to the user. For **overhanging** lift-off: front/left/right extension is correctly independent and additive (fixture: `overhang.metrics.dimensions.lidWidth===outsideWidth+left+right`), **rear overhang is correctly forced to zero** (`values.overhangRear=0` unconditionally in `normalizeDesignDraft`, fixture-confirmed: `normalizeDesignDraft({...base,giftLidAppearance:'overhanging'}).values.overhangRear===0`), and the locating frame's opening formula does not depend on lid appearance (frame remains keyed to `insideWidth/insideDepth`, unaffected by overhang) — correct.

## 7. Hinged-lid findings

Structural data (hinge positions, screw offsets, back/lid leaf-guide coordinates) is computed by dedicated, well-validated helper functions (§8) and is independent of the Finished View. The **Closed** Finished View's claim that the lid sits horizontal/seated cannot currently be assessed because the lid does not render at all (§11 — the identical NaN-coordinate bug affects hinged mode exactly as it affects lift-off mode, since `buildGiftBoxFinishedViewSvg`'s lid-point construction is shared and does not branch on `lidType`). The **Open** view's claim of "approximately 90 degrees" is **not represented by any actual rotation of the lid polygon** — even setting aside the NaN bug, tracing the code shows `lidPts` for `state==='open'` differs from `state==='closed'` only by position (`lidX=150,lidY=42` vs `lidX=x+w+18,lidY=y-h-dy-10`) and a uniform `lidScale=.78`; **no rotation transform or angle-dependent point recomputation exists anywhere in this function**. This is a second, independent defect layered on top of the rendering failure — even after fixing the NaN issue, the "open" state would show the lid as a smaller, repositioned flat rectangle, not a rotated one, directly contradicting the on-screen caption text "Open lid approximately 90 degrees" (§20 — this exact caption is the *only* thing the fixture suite checks). Rear-overhang-zero and frame-defaults-off are correctly enforced structurally (`values.frameEnabled && values.lidType==='hinged-surface'` is a **hard error** — "Locating frame conflicts with surface-mounted hinge guides; disable the locating frame for hinged lids" — fixture-confirmed: `!buildDesignResult({...base,giftLidType:'hinged-surface'}).valid` when frame is left enabled). ## 10. Finished Closed findings — **confirmed defect**

Live-reproduced with the user's exact reported inputs (120×90×50 mm internal, 2.88 mm thickness, frame enabled): the rendered `<polygon id="gift-box-lid">` element's `points` attribute is literally **`"NaN,NaN NaN,NaN NaN,NaN NaN,NaN"`**. Per the SVG specification, a `<polygon>` with non-numeric point values renders as **nothing** — it is completely invisible. This single defect, by itself, explains complaint #1 ("does not visually read as a closed box" — because there is no visible lid) and is the underlying cause of #2/#5 (the LID text label is positioned using the same `lidX/lidY/lidW/lidD` scalars, which are numerically valid, but the label now points at an empty region, since the polygon it's supposed to label never renders — hence "the LID label is visually disconnected from the apparent lid plane"). **Root cause, confirmed by direct source read** (`buildGiftBoxFinishedViewSvg`): `lidPts` is constructed as an array of **plain 2-element arrays** — `lidPts=[[lidX,lidY],[lidX+lidW,lidY],...]` — but the shared `poly(id,cls,pts,extra)` rendering helper used for every other shape in this same function (`front`,`side`,`back`,`cavity`) expects an array of **`{x,y}` objects** (`pts.map(p=>\`${designRound(p.x)},${designRound(p.y)}\`)`). A plain array has no `.x`/`.y` properties, so `p.x`/`p.y` evaluate to `undefined`, and `designRound(undefined)` returns `NaN`. This is a straightforward data-shape mismatch, not a geometric-modeling error. The frame rectangles were live-confirmed positioned at `x≈95.4, y≈124.6` (near the *box's own* top-back-corner region, per the `back`/`front` polygon coordinates captured in the same session), entirely disconnected from wherever a correctly-rendered lid would sit (which, per the closed-state formula, would be at `x+w+18 ≈ 258`, well to the right) — this independently confirms complaint #4/#7 ("frame lines appear to float above or beside the structure," "frame is not clearly shown as rails attached to the underside of the lid").

## 11. Finished Open findings — **confirmed defect**

The identical NaN-coordinate bug applies to the Open state (live-confirmed: `openLidRect` also `"NaN,NaN NaN,NaN NaN,NaN NaN,NaN"`), so the lid is equally invisible in Open. Additionally — **independent of the NaN bug** — no rotation is ever computed for the lid between Closed and Open (§7): the two states differ only in the lid's flat-projected position and a uniform 0.78 scale factor, never an angle. For **lift-off** mode specifically, since `front`/`side`/`back`/`cavity`/`frame` are all state-independent and the lid is invisible in both states, **Closed and Open render visually identical except for caption text** — this precisely and completely explains complaint #3 ("Finished Open appears very similar to Finished Closed"). For hinged mode, the hinge markers *do* depend on `view.hingePositions` but not on `state` (they render identically in both Closed and Open), so even in hinged mode there is no visual indication of an opening rotation beyond the lid's (currently invisible) reposition/rescale.

## 13. Assembly-label warning findings

Live-reproduced exactly: enabling assembly labels on the default (lift-off, frame-on) configuration produces:
```
Warning: Assembly label FRAME-LEFT omitted: no safe polygon placement on FRAME-LEFT.
Warning: Assembly label FRAME-RIGHT omitted: no safe polygon placement on FRAME-RIGHT.
Warning: Assembly label HINGE-COUPON omitted: missing panel hinge-coupon.
```
1. **Are the rails genuinely too narrow?** Yes — at the default 8 mm rail width, the shared safe-label-placement algorithm (unmodified, pre-existing) correctly finds no safe interior polygon for a multi-character label; this is consistent, expected behavior for a narrow panel, not a bug in the label-safety logic.
2. **Do the pieces remain identifiable without labels?** Yes, and more strongly: `frame-left` and `frame-right` are **geometrically identical rectangles** (§9), so they are physically interchangeable — omitting their labels causes **zero real assembly ambiguity** (either rail can go on either side).
3. **Could labels be handled better?** Yes — shortening to "FL"/"FR", using a leader line outside the panel, or omitting the attempt entirely for known-narrow/interchangeable rails are all reasonable, bounded options.
4. **Is omission expected for valid default settings?** Yes — this occurs on the *default* configuration whenever labels are enabled.
5. **Should it be informational rather than a warning?** Given the pieces are provably interchangeable, **yes** — the current "Warning:" framing overstates the practical risk.
6. **Does it make a valid design look broken?** Very likely yes, and it directly compounds the Finished View problem in the user's overall impression.

**A third, previously-unreported defect was found and reproduced live:** the "HINGE-COUPON omitted: missing panel hinge-coupon" warning fires **even in lift-off mode**, where a hinge coupon was never configured or expected to exist. Tracing `designAssemblyLabelSpecs`'s `gift` branch confirms it **unconditionally** returns a fixed 10-label list including `{panelId:'hinge-coupon', text:'HINGE-COUPON'}` regardless of `values.lidType`/`values.hingeCoupon` — the list is never filtered to only the panels that actually exist for the current configuration. This produces a nonsensical, confusing warning about a part that was never intended to be present. **Classified IMPORTANT** (a clear, avoidable, confusing default-configuration warning, distinct from the narrow-rail question). **Smallest fix:** condition the `gift` label list on `values.frameEnabled` (for the frame-rail labels) and `values.lidType==='hinged-surface' && values.hingeCoupon` (for the hinge-coupon label) inside `designAssemblyLabelSpecs`, so only labels for panels that actually exist are attempted; separately, downgrade the FRAME-LEFT/FRAME-RIGHT omission to informational wording (or shorten the label text) given the confirmed interchangeability of those two rails.

## 19. Fixture-quality findings

All 60 `runGiftBoxFixtures` assertions were individually mapped. The large majority are genuinely behavioral (directly invoking `buildDesignResult`/`normalizeDesignDraft` with realistic drafts and checking numeric/structural outcomes) and correctly cover: default validity, dimension preservation, one-thickness-drives-shell-and-lid, panel-ID explicitness, finger-pattern parity, frame defaults/opening math, flush-lid-matches-footprint, overhang-changes-only-lid, external/inch dimension modes, kerf/fit reporting, hinge-mode validity and all its blocking-validation branches, screw-pattern centering, panel-spacing effect on layout, closed-contour check, SVG determinism, red-cut/blue-guide group presence, and Project-handoff behavior. **However — and this is the central fixture-quality finding of this audit — every assertion that touches the Finished View is a source-string or attribute-presence check, never a spatial/coordinate verification:**
- `'closed Finished View identifies three-quarter state'` — checks the *caption text* "Closed three-quarter view" is present.
- `'open Finished View identifies approximate 90 degree state'` — checks the *caption text* "approximately 90 degrees" is present.
- `'open and closed Finished Views differ'` — checks the two SVG strings are not byte-identical (true, but only because of a scale factor/position shift and captions — not because a correct rotation occurred).
- `'Finished View reports locating frame'` — checks the `data-frame-enabled="true"` attribute is present.
- `'Finished View reports lid appearance'` — checks the `data-lid-appearance="overhanging"` attribute is present.

**None of these five assertions would fail if the lid polygon were entirely invisible (NaN) and never rotated** — which is exactly the confirmed live state. This is precisely why a 60/60-passing fixture suite shipped with a completely broken lid rendering. This is a **serious, confirmed fixture-quality gap** requiring correction before commit, independent of fixing the underlying rendering bug itself (a fix to the bug without adding a coordinate-level regression test would leave the suite unable to prevent recurrence).

## 26. Findings by severity

**BLOCKER (0):** none. Production dimensions, finger-joint ownership, contour closure, SVG determinism/validity, and existing-output regression are all confirmed correct. **IMPORTANT (3):**
- **GB-I1 — Finished View lid is completely invisible in both Closed and Open states**, caused by a data-shape mismatch (`lidPts` built as `[x,y]` arrays instead of the `{x,y}` objects the shared `poly()` renderer requires), producing `points="NaN,NaN NaN,NaN NaN,NaN NaN,NaN"`. This is the confirmed root cause of essentially all nine user-reported visual complaints (§12). **Smallest fix:** in `buildGiftBoxFinishedViewSvg`, construct `lidPts` as an array of `{x,y}` objects matching the `point()`-helper convention used for every other shape in the same function (e.g., `lidPts=[{x:lidX,y:lidY},{x:lidX+lidW,y:lidY},{x:lidX+lidW+lidD*.34,y:lidY+lidD*.34},{x:lidX+lidD*.34,y:lidY+lidD*.34}]`). Storage/schema-neutral; does not touch production SVG bytes (the bug is confined to `buildGiftBoxFinishedViewSvg`, confirmed isolated in §15).
- **GB-I2 — No rotation is applied to the lid between Closed and Open states**, even independent of GB-I1. The Open caption claims "approximately 90 degrees" but the lid geometry only repositions and uniformly scales by 0.78 — a materially misleading Finished View per the audit's own explicit standard. **Smallest fix:** for `lidType==='hinged-surface'`, compute the Open-state lid points via an actual rotation (e.g., project the lid plane about the back-top hinge-axis line at a fixed ~75–90° apparent angle in the same isometric projection used elsewhere in the function) rather than a flat reposition/rescale; for `lidType==='lift-off'`, the current "moved aside" concept is conceptually acceptable per the audit's own guidance but must be visually legible once GB-I1 is fixed (a lid rendered clearly to one side, at full/near-full scale rather than 0.78, reads more clearly as "removed" than as a distant, shrunken copy).
- **GB-I3 — Fixture suite cannot detect either GB-I1 or GB-I2.** All Finished-View assertions are string/attribute checks (§19), none inspect actual coordinates or verify absence of `NaN`. **Smallest fix:** add assertions that parse the returned SVG (e.g., via `DOMParser`, as `runGiftBoxFixtures` already does for the production SVG) and check `#gift-box-lid`'s `points` attribute contains no `NaN` and its bounding box is non-degenerate and spatially plausible relative to the box body, for both Closed and Open states.

Additionally, bundled into GB-I3's remediation scope but independently confirmed as its own finding: **GB-I4 — the "HINGE-COUPON omitted" warning fires in lift-off mode where no hinge coupon was ever configured**, and **the FRAME-LEFT/FRAME-RIGHT omission warning uses alarming wording for a case where the two rails are confirmed geometrically identical/interchangeable** (§13). **Smallest fix:** condition the Gift Box assembly-label spec list on which panels actually exist for the current configuration, and consider downgrading the frame-rail label omission to informational wording or a shortened label (FL/FR) given the confirmed zero-risk interchangeability.

## 27. Exact final verdict
